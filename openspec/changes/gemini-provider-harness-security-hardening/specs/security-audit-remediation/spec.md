## ADDED Requirements

### Requirement: 系统 SHALL 建立风险分级与整改时限机制
系统 MUST 将安全审计结果按 P0/P1/P2 分级，并为每个风险等级定义统一整改时限、责任人和复验要求，确保整改过程可追踪。

#### Scenario: 产生审计结果后自动分级
- **WHEN** 代码安全审计任务输出漏洞与风险项
- **THEN** 系统 MUST 生成包含风险级别、影响范围、整改建议、责任归属与截止时间的结构化记录

#### Scenario: 超时整改触发升级
- **WHEN** 任一风险项超过定义时限仍未关闭
- **THEN** 系统 MUST 触发升级流程并阻断相应发布路径，直到完成复验

#### Scenario: 分级整改时限明确
- **WHEN** 安全治理基线生效
- **THEN** 系统 MUST 使用统一整改时限：P0 ≤ 24 小时、P1 ≤ 7 天、P2 ≤ 30 天

### Requirement: 系统 SHALL 强制执行高危风险阻断策略
对于 P0 高危项（如密钥泄露、误实盘风险、鉴权失效），系统 MUST 在 CI/发布前提供阻断式校验，不得以仅告警方式放行。

#### Scenario: 检测到疑似凭据泄露
- **WHEN** pre-commit 或 CI 检测到凭据泄露模式
- **THEN** 系统 MUST 阻断提交/合并并提供轮换与处置指引

#### Scenario: 实盘安全闸门不满足
- **WHEN** 交易执行路径未满足实盘确认前置条件
- **THEN** 系统 MUST 拒绝下单执行并记录安全日志

#### Scenario: 实盘安全闸门前置条件定义
- **WHEN** 系统判定允许进入实盘下单
- **THEN** 必须同时满足 `TEST_MODE=false`、显式实盘确认开关、凭据完整通过校验、运行环境处于受控实盘环境

### Requirement: 系统 SHALL 具备密钥轮换与处置留痕能力
系统 MUST 为密钥泄露处置提供标准化流程，至少包括吊销、轮换、验证、回滚和审计记录。

#### Scenario: 完成密钥轮换处置
- **WHEN** 团队执行密钥轮换流程
- **THEN** 系统 MUST 留存包含时间线、影响资产、验证结果与回滚状态的处置记录

### Requirement: 系统 SHALL 提供安全复验闭环
每个已修复风险项 MUST 具备可重复执行的复验方法（命令/测试/检查项），并在审计报告中标记复验通过状态。

#### Scenario: 风险修复后执行复验
- **WHEN** 开发者提交风险修复变更
- **THEN** CI MUST 执行对应复验步骤并更新风险状态为“已验证关闭”或“复验失败”

#### Scenario: P0 风险必须阻断
- **WHEN** 检测到 P0 风险（如密钥泄露、误实盘、鉴权失效）
- **THEN** 系统 MUST 立即阻断合并与发布，不得进入仅告警模式
